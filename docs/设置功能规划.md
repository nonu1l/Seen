# 设置功能规划

本文档规划 Seen 的设置功能。第一阶段只覆盖“AI 助手”和“搜索与数据源”，其他展示偏好、备份、统计、认证等设置后续再扩展。

## 目标

- 在前端提供独立设置页 `/settings`。
- 支持查看和修改 AI 助手相关配置。
- 支持查看和修改搜索源、Serper、Bangumi 代理等数据源配置。
- 提供连接测试能力，减少用户配置后只能靠报错排查的问题。
- 所有设置保存后立即生效，避免 Docker 部署场景下频繁重启容器。

## 当前现状

当前配置主要来自 `application.yml` 和环境变量：

- `seen.ai.enabled`
- `spring.ai.openai.api-key`
- `spring.ai.openai.base-url`
- `spring.ai.openai.chat.options.model`
- `spring.ai.openai.chat.options.temperature`
- `seen.search.provider`
- `seen.search.serper-api-key`
- `seen.bangumi-proxy`
- `seen.detail.cast-enabled`

当前代码中不少配置通过 `@Value` 在 Bean 创建时注入，例如：

- `AiFeatureGuard` 使用 `seen.ai.enabled`
- `AppConfigController` 暴露 `seen.ai.enabled`
- `WebSearchService` 启动时固定选择 Serper 或 DuckDuckGo
- `SerperSearchService` 启动时读取 Serper API Key
- `DDGSearchService` 启动时基于 Bangumi 代理拼出搜索地址
- `BangumiService` / `PreCacheService` 启动时固定 Bangumi API base URL

因此第一阶段不能继续沿用启动时固定配置的设计。设置功能落地时，需要同步把相关服务改造成运行时读取配置，确保保存后立即生效。

## 第一阶段范围

### AI 助手

设置项：

- AI 助手开关
- API Base URL
- API Key
- 模型名称
- Temperature
- 是否记录 token 使用明细

操作：

- 保存配置
- 测试 AI 连接
- 隐藏 / 显示 API Key
- 保存后立即生效

第一阶段建议：

- `aiEnabled` 必须运行时生效。
- Spring AI 的 `base-url`、`api-key`、`model`、`temperature` 必须保存后立即生效。
- 会话调用时从 `SettingsService` 获取当前配置，并通过动态 `ChatClient` 工厂创建或选择客户端。
- 已经进行中的单次 AI 请求不用中断；下一次请求使用最新配置。

### 搜索与数据源

设置项：

- 搜索源：`serper` / `ddg`
- Serper API Key
- Bangumi 代理地址
- 是否展示角色 / 演员信息

操作：

- 保存配置
- 测试 Serper 搜索
- 测试 DuckDuckGo 搜索
- 测试 Bangumi API
- 测试图片代理可用性

第一阶段建议：

- 搜索源切换必须保存后立即生效。
- Bangumi 代理地址必须保存后立即生效。
- 角色 / 演员信息开关必须保存后立即生效。
- 服务不应在构造函数中缓存 provider、API Key 或 base URL。
- 测试接口应使用用户输入的临时配置值，不依赖已经保存或已经生效的 Bean。

## 信息架构

设置页使用两栏布局：

- 桌面端：左侧设置分组，右侧当前分组内容。
- 移动端：顶部横向 tabs，下面显示当前分组内容。

第一阶段只展示两个分组：

- AI 助手
- 搜索与数据源

页面顶部：

- 标题：`seen. settings`
- 返回按钮：回到上一页，无法回退时回首页
- 保存按钮：有未保存变更时高亮
- 未保存提示：离开页面前提示确认
- 保存成功提示：设置已生效

## 前端设计

### 设置入口

在全局 header 右侧增加设置按钮：

- 首页：显示齿轮按钮，点击进入 `/settings`
- AI 页：保留退出、重置按钮，同时显示齿轮按钮
- 设置页：显示返回按钮，不再重复显示齿轮按钮

按钮样式沿用现有 `btn-icon`，使用齿轮图标。

### AI 助手表单

控件建议：

- AI 助手开关：toggle
- API Base URL：文本输入框
- API Key：密码输入框，右侧显示 / 隐藏按钮
- 模型名称：文本输入框，后续可扩展为下拉选择
- Temperature：数字输入或滑块，范围 `0` 到 `2`
- Token 记录：toggle
- 测试 AI：按钮

辅助提示：

- Base URL 示例：`https://api.deepseek.com`
- OpenAI 兼容模型需要支持 `/chat/completions`
- 保存成功后，下一次 AI 请求立即使用新配置

### 搜索与数据源表单

控件建议：

- 搜索源：segmented control，选项为 `Serper`、`DuckDuckGo`
- Serper API Key：密码输入框
- Bangumi 代理地址：文本输入框
- 展示角色 / 演员信息：toggle
- 测试搜索：按钮
- 测试 Bangumi：按钮

辅助提示：

- DuckDuckGo 当前通过 Bangumi 代理的 `/search` 路径访问。
- Serper 需要 API Key。
- Bangumi 代理为空时回退到 `https://api.bgm.tv/v0`。

## 后端设计

### 配置持久化

新增配置表：

```sql
app_setting
- id
- setting_key
- setting_value
- value_type
- sensitive
- updated_at
```

约束：

- `setting_key` 唯一。
- 敏感字段如 API Key 保存时应标记 `sensitive = true`。
- 返回给前端时，敏感字段默认脱敏。
- 所有设置保存后立即生效，不设计 `restart_required` 字段。

建议 key 命名：

```text
seen.ai.enabled
seen.ai.token-usage-enabled
spring.ai.openai.base-url
spring.ai.openai.api-key
spring.ai.openai.chat.options.model
spring.ai.openai.chat.options.temperature
seen.search.provider
seen.search.serper-api-key
seen.bangumi-proxy
seen.detail.cast-enabled
```

### 配置读取优先级

建议统一为：

```text
数据库设置 > 环境变量 / application.yml > 代码默认值
```

新增 `SettingsService` 统一读取配置，避免业务代码散落读取 `@Value`。

第一阶段可以先做到：

- `SettingsService` 读取数据库设置。
- 如果数据库没有值，则读取当前 Spring Environment。
- `GET /api/settings` 返回最终生效值和来源。

### DTO 设计

`SettingsResponse`：

```json
{
  "groups": [
    {
      "key": "ai",
      "label": "AI 助手",
      "settings": [
        {
          "key": "seen.ai.enabled",
          "label": "AI 助手",
          "value": true,
          "type": "boolean",
          "sensitive": false,
          "effectiveSource": "database"
        }
      ]
    }
  ]
}
```

`UpdateSettingsRequest`：

```json
{
  "settings": {
    "seen.ai.enabled": true,
    "spring.ai.openai.base-url": "https://api.deepseek.com",
    "seen.search.provider": "serper"
  }
}
```

测试请求：

```json
{
  "baseUrl": "https://api.deepseek.com",
  "apiKey": "sk-xxx",
  "model": "deepseek-v4-flash"
}
```

## API 规划

### 读取设置

```http
GET /api/settings
```

返回：

- 设置分组
- 当前值
- 是否敏感
- 当前值来源：`database` / `environment` / `default`

### 保存设置

```http
PUT /api/settings
```

行为：

- 只保存允许编辑的 key。
- 敏感字段如果传空字符串，不覆盖原值。
- 保存后返回最新设置。
- 保存后立即影响后续请求。

### 测试 AI

```http
POST /api/settings/test-ai
```

行为：

- 使用请求体中的临时配置调用模型。
- 发送极短测试 prompt。
- 返回连接是否成功、模型名、错误信息。

### 测试搜索

```http
POST /api/settings/test-search
```

行为：

- 使用请求体中的搜索源和 API Key。
- 查询固定关键词，例如 `孤独摇滚`。
- 返回命中数量和前几条标题。

### 测试 Bangumi

```http
POST /api/settings/test-bangumi
```

行为：

- 使用请求体中的 Bangumi 代理地址。
- 请求 `/api/subjects/1` 或轻量搜索接口。
- 返回是否成功、响应耗时、错误信息。

## 运行时生效策略

| 设置项 | 第一阶段 | 后续优化 |
|---|---|---|
| AI 助手开关 | 可运行时生效 | 保持 |
| Token 记录开关 | 可运行时生效 | 保持 |
| API Base URL | 保存后立即生效 | 可增加客户端缓存 |
| API Key | 保存后立即生效 | 可增加客户端缓存 |
| Model | 保存后立即生效 | 可增加客户端缓存 |
| Temperature | 保存后立即生效 | 可增加客户端缓存 |
| 搜索源 | 保存后立即生效 | 保持 |
| Serper API Key | 保存后立即生效 | 保持 |
| Bangumi 代理 | 保存后立即生效 | 可按 base URL 缓存短期客户端 |
| 角色 / 演员开关 | 保存后立即生效 | 保持 |

实现原则：

- 可变配置不能只通过 `@Value` 注入到字段后长期缓存。
- `@Value` / `Environment` 只作为默认值来源。
- 业务服务每次执行请求前，从 `SettingsService` 读取当前设置。
- `SettingsService` 可以做内存快照缓存，但保存设置后必须同步刷新缓存。
- 对 API Key、base URL 这类配置，可按配置 hash 缓存客户端实例；保存设置后清理相关缓存。

## 实施步骤

### 第一步：文档与路由

- 新增 `/settings` 路由。
- 在 header 增加设置按钮。
- 新增设置页基础布局。
- 第一版前端使用静态表单结构。

### 第二步：后端设置读取

- 新增 `AppSetting` 实体。
- 新增 `AppSettingRepository`。
- 新增 `SettingsService`。
- 新增 `SettingsController`。
- 实现 `GET /api/settings`。
- 将环境变量和 `application.yml` 作为默认值导入展示。

### 第三步：后端设置保存

- 实现 `PUT /api/settings`。
- 敏感字段脱敏展示。
- 敏感字段空值不覆盖。
- 保存后刷新 `SettingsService` 内存快照。
- 返回“已生效”状态。

### 第四步：连接测试

- 实现 `POST /api/settings/test-ai`。
- 实现 `POST /api/settings/test-search`。
- 实现 `POST /api/settings/test-bangumi`。
- 前端显示测试结果。

### 第五步：保存即生效改造

- `AiFeatureGuard` 改为读取 `SettingsService`。
- `AppConfigController` 改为读取 `SettingsService`。
- Token 用量记录增加开关判断。
- `WebSearchService` 每次请求时按 `SettingsService` 选择 provider。
- `SerperSearchService` 每次请求时读取当前 API Key。
- `DDGSearchService` 每次请求时读取当前代理地址。
- `BangumiService` 每次请求时读取当前代理地址并生成 base URL。
- AI 会话调用改为动态构造或选择 `ChatClient`。

### 第六步：客户端缓存优化

- 为动态 `ChatClient` 增加按配置 hash 复用机制。
- 为搜索和 Bangumi 请求客户端增加轻量缓存。
- 保存相关设置后清理对应客户端缓存。

## 前端状态处理

- 页面加载时请求 `GET /api/settings`。
- 用户修改表单后进入 dirty 状态。
- 保存成功后清除 dirty 状态。
- 保存成功后显示提示：“设置已生效”。
- 测试按钮独立 loading，不影响保存按钮。
- API Key 类字段默认显示为 `********`，用户输入新值后才提交覆盖。

## 安全与隐私

- API Key 不在列表接口中明文返回。
- 日志中不打印 API Key。
- 测试接口异常返回应隐藏完整密钥和请求头。
- 后续如果部署公网，应优先增加登录或管理员 token，再开放设置写接口。

## 验收标准

- 用户可以从首页和 AI 页进入设置页。
- 用户可以查看 AI 助手、搜索与数据源两组配置。
- 用户可以保存配置。
- 用户保存配置后，后续请求立即使用新配置，不需要重启 Docker 容器。
- 用户可以测试 AI、搜索、Bangumi 连接。
- API Key 不会在前端初始化时明文展示。
- 关闭 AI 后，前端隐藏 AI 入口，后端拒绝 AI 会话接口。

- 展示偏好
