# AGENTS.md

## 基本规则

- 完成代码改动后需要提交代码；提交时只包含本次任务相关文件，不要带入用户已有的未提交改动。
- 不需要主动编译 Java 代码；如需验证，优先运行前端构建、静态检查或不改写源码的测试命令。
- 代码改动完成后，需要同步维护 `README.md` 的版本记录；如果当前版本记录已有可合并条目，直接合并补充，不重复新增零散记录。
- 纯文档说明、规则文本或内部配置整理，不强制新增版本记录；用户明确要求记录时除外。

## 注释要求

- 新增类和新增方法需要补充简洁注释，说明用途、关键参数或副作用。
- 修改现有方法后，如果原注释已经不准确，需要同步更新注释。
- 注释应解释业务意图或不明显的约束，避免重复描述代码表面行为。

## 配置提取规则

- 适合放入 `application.yml` 的参数：部署环境相关 endpoint、运行超时、重试次数、缓存大小、缓存 TTL、上下文长度、返回数量上限、Agent 安全阈值、Web 抓取字节/字符上限等。
- 不适合放入 `application.yml` 的参数：用户会在前端设置页调整的 AI Base URL、API Key、模型、temperature、思考模式、搜索源、搜索 API Key、Bangumi 代理、详情开关等；这些应继续走设置页和数据库。
- Prompt 内容应放在 `src/main/resources/prompts/*.st`，不要放进 YAML；业务枚举、实体字段、DTO 字段、解析正则和 UI 文案也不要为了“可配置”而迁入 YAML。
- 新增可配置项时，参数原本属于哪个服务就在哪个服务里定义本地 `final` 字段，并通过构造函数 `@Value("${配置项:默认值}")` 从 `application.yml` 注入；同时在 `application.yml` 写出当前默认值，确保配置缺失时行为与改动前一致。
- 提取配置前先判断它是否属于“运行策略/部署调优”，不要把只服务于代码结构的常量、内部 cache key、ThreadLocal 名称、日志对象等抽成配置。

## 工作区约束

- 允许工作区存在用户未提交改动；除非用户明确要求，不要回滚或格式化这些无关改动。
- 提交前检查 `git status`，确认暂存内容只覆盖当前任务。
